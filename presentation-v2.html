OOP → Spring · All 4 Pillars in Our Code

You already knew this. Spring just uses it.

🔒

Encapsulation

PrepSession.java — private fields + getters/setters. Spring & JPA call your getters to read/write the DB. Not optional.

🧬

Inheritance

PrepSessionRepository extends JpaRepository — inherits save(), findAll(), delete() for free. Zero implementation needed.

🎭

Abstraction

JpaRepository is an interface — you call save() without writing a single line of SQL. The how is hidden. You only need the what.

🔀

Polymorphism

PrepController uses PrepService by type — Spring decides the concrete object at runtime. You code to the interface, not the implementation.

OOP isn't just theory. Every pillar shows up in the 4 files you're about to write.

Iteration 4 of 6

Save it. Zero SQL written.

One annotation. One interface. Full database support.

Add to PrepSession.java
@Entity // ← this one line public class PrepSession { @Id @GeneratedValue private Long id; private String topic; private String questions; // getters, setters }
New file: PrepSessionRepository.java
public interface PrepSessionRepository extends JpaRepository<PrepSession, Long> { } // That's it. Seriously. // save() is already there ✅
Spring sees @Entity → creates the table. Spring sees the interface → generates all SQL. You wrote zero SQL.

The Spring Magic — DI · Annotations · Layers

You annotate. Spring creates. You receive.

Dependency Injection

❌ Old: new PrepService() — you manage the object
✅ Spring: PrepService service in constructor — handed to you

🍕 You don't grow wheat to make pizza. Zomato delivers. Spring delivers.

Annotations → Beans

@Service = business logic  ·  @Repository = data access
@RestController = HTTP  ·  @Entity = DB table

Spring reads these at startup, creates objects, injects where needed. You never call new. Ever.

🧑‍💼ControllerReceives the HTTP request — nothing else
↓
👨‍🍳ServiceProcesses business logic — calls AI, applies rules
↓
🗄️RepositoryTalks to the database — save, fetch, delete

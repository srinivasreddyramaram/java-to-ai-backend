OOP → Spring

Encapsulation

"Hide the data. Control the access." — PrepSession.java in our code
❌ No encapsulation
Java
public class PrepSession { public String topic; public String questions; // anyone can do: // session.topic = ""; // no control 😬 }
✅ PrepSession.java — our code
Java
private String topic; // hidden public String getTopic() { return topic; // read ✅ } public void setTopic(String t) { this.topic = t; // write ✅ }
Spring & JPA call your getters/setters to read and write the database. Encapsulation is not just theory — it's required.
6 / 16
OOP → Spring

All 4 OOP Pillars — In Our Code

🔒 Encapsulation PrepSession.java — private fields + getters/setters
🧬 Inheritance PrepSessionRepository extends JpaRepository — inherits save, findAll, delete for free
🎭 Abstraction JpaRepository is an interface — you call save() without writing a single line of SQL
🔀 Polymorphism PrepController uses PrepService by type — Spring decides the concrete object at runtime
OOP isn't just theory. Every pillar shows up in the 4 files you're about to write.

Concept 1

Dependency Injection

"Stop newing, start receiving"
❌ Plain Java
Java
PrepService service = new PrepService(); // you manage the object // you manage lifecycle // you manage everything
✅ Spring
Java
public PrepController( PrepService service) { this.service = service; // handed to you ✅ }
🍕 You don't grow wheat to make pizza. Zomato delivers. Spring delivers.

Concept 2

Annotations → Beans

"You annotate. Spring creates. You receive."
📌 You write — annotation
Java
@Service public class PrepService { // your logic here } @Repository public class PrepRepo { // your DB code }
→
🫘 Spring does — creates bean
// reads annotation at startup // creates the object // stores in container // injects where needed You never call new. Ever.
@Service  business logic  ·  @Repository  data access  ·  @RestController  HTTP  ·  @Entity  DB table
